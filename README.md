# یوزر منیجر یکپارچه (MikroTik: WireGuard/OpenVPN/L2TP + V2Ray/Xray + ربات تلگرام)

پنل مدیریت کاربران VPN با یک **سهمیه (حجم) مشترک** برای هر کاربر؛ هر کاربر می‌تواند هم‌زمان چند اتصال از پروتکل‌های مختلف داشته باشد (WireGuard، OpenVPN و L2TP روی میکروتیک، و/یا یک اکانت V2Ray/Xray) و مصرف همه از همان یک سهمیه کم می‌شود. وقتی حجم تمام شود یا تاریخ انقضا برسد، تمام اتصالات آن کاربر به‌صورت خودکار غیرفعال می‌شوند.

علاوه بر پنل مدیریت، این پروژه شامل موارد زیر هم هست:

- یک **ربات تلگرام کامل** (فروش/پشتیبانی/مدیریت) که مستقیم داخل همین بک‌اند اجرا می‌شود - بدون سرور جدا، بدون فایل `.env` دستی؛ همه چیز از پنل تنظیم می‌شود.
- سیستم **پکیج‌های قابل‌فروش** (حجم + مدت + قیمت + سرویس‌های همراه + فایل/متن دلخواه) که هم از پنل و هم از ربات قابل استفاده‌اند.
- **کیف پول اعتباری** برای هر مشتری (شارژ با کارت‌به‌کارت + تایید ادمین، یا پرداخت آنی از موجودی).
- بخش **آموزش** (متن + عکس/ویدیو) قابل مدیریت از پنل و قابل مشاهده از ربات.
- **بک‌آپ خودکار دیتابیس** (۴ بار در روز) + بک‌آپ دستی، هر دو با ارسال خودکار به ادمین‌های ربات در تلگرام.
- امکان **نصب ربات روی یک سرور دوم** (مثلا برای IP تمیزتر یا پوشش جغرافیایی بهتر) در حالی که همچنان به دیتابیس واقعی همین سرور وصل می‌ماند.

یک **API جداگانه برای ربات/سیستم فروش شخص ثالث** هم وجود دارد که می‌توانید بدون لاگین ادمین، کاربر بسازید، تمدید کنید یا وضعیتش را چک کنید (بخش «API ربات مشتری» را ببینید) - همان API که ربات داخلی خودِ پروژه هم از آن استفاده می‌کند.

## معماری خلاصه

- هر **کاربر** یک `total_quota_bytes` و `used_bytes` مشترک دارد، به‌علاوه یک `balance` (کیف پول، تومان).
- هر کاربر می‌تواند چند **اتصال (Connection)** داشته باشد: `wireguard` / `openvpn` / `l2tp` (همه روی یک نود میکروتیک) و/یا `xray` (روی یک نود V2Ray).
- یک **پروسه پس‌زمینه** (هر ۳۰ ثانیه، قابل تنظیم) به تمام نودها سر می‌زند، مصرف جدید هر اتصال را می‌گیرد، دلتای آن را به `used_bytes` کاربر اضافه می‌کند و اگر سهمیه/انقضا تمام شده باشد، اتصال را در همان لحظه در میکروتیک/Xray غیرفعال می‌کند.
- **ربات تلگرام** یک ترد پس‌زمینه جدا با event loop خودش است (مثل سرور RADIUS) و از طریق `panel_bridge.py` مستقیم با دیتابیس پنل صحبت می‌کند - هیچ HTTP round-trip اضافه‌ای در حالت عادی وجود ندارد. اگر ربات روی سرور دوم نصب شود، همین لایه پل خودکار به حالت HTTP (با `X-API-Key`) سوییچ می‌کند، بدون این‌که کد خودِ ربات (handlers) تغییری نیاز داشته باشد.
- تنظیمات هر سرور (شامل پورت API میکروتیک) از صفحه «سرورها» قابل ویرایش است؛ نیازی به حذف و ساخت دوباره نیست.

## پیش‌نیازها

### ۱. میکروتیک

- RouterOS **نسخه ۷ به بالا**.
- یک کاربر API با دسترسی کافی (Read/Write) در `System > Users`.
- پورت API (پیش‌فرض `8728`) و پورت API-SSL (پیش‌فرض `8729`) هر دو مستقل از هم و از صفحه «سرورها» قابل تغییرند؛ کدام‌یک استفاده شود با تیک «استفاده از API-SSL» مشخص می‌شود.

**WireGuard:** اینجا پنل همه‌چیز را خودش مدیریت می‌کند - اینترفیس، کلیدها، IP کلاینت و peer، همه از طریق API میکروتیک ساخته می‌شوند. اگر اینترفیس از قبل نساخته‌اید، پنل خودش موقع اولین اتصال آن را می‌سازد؛ فقط IP عمومی روتر (Endpoint) و پورت WireGuard را درست وارد کنید. یک بلاک IP خالی برای کلاینت‌ها در نظر بگیرید (پیش‌فرض `10.66.66.0/24`).

**OpenVPN و L2TP - احراز هویت با RADIUS:** برای این دو پروتکل، پنل **فقط یوزرنیم/پسورد** می‌سازد و از طریق یک **سرور RADIUS واقعی که خودِ پنل اجرا می‌کند** آن‌ها را تایید می‌کند (دیگر از `/ppp/secret` محلی روی روتر استفاده نمی‌شود). هر چیز دیگری - IP pool، خود سرویس OpenVPN-server/L2TP-server، سرتیفیکیت OpenVPN، و تنظیمات IPsec - باید از قبل و مستقیما روی خود میکروتیک (Winbox/CLI) تنظیم شده باشد؛ پنل به آن‌ها دست نمی‌زند و آن‌ها را overwrite نمی‌کند. کارهایی که باید یک‌بار خودتان روی روتر انجام دهید:

1. یک IP Pool برای کلاینت‌های OpenVPN/L2TP بسازید و آن را در پروفایل PPP پیش‌فرض تنظیم کنید.
2. برای OpenVPN: یک CA + یک سرتیفیکیت سرور بسازید (`System > Certificates`) و در `/interface/ovpn-server/server` فعالش کنید.
3. برای L2TP: `/interface/l2tp-server/server` را فعال و در صورت نیاز IPsec/PSK را تنظیم کنید.
4. روتر را طوری تنظیم کنید که برای احراز هویت PPP از RADIUS این پنل استفاده کند (بخش «راه‌اندازی RADIUS» زیر را ببینید).

در تنظیمات نود در پنل، فیلدهای «RADIUS Secret»، «پورت OpenVPN»، «نام سرتیفیکیت» و «کلید IPsec» را هم پر کنید - پورت/سرتیفیکیت/IPsec را پنل به روتر پوش نمی‌کند، فقط برای این ذخیره می‌شوند که در متنی که به مشتری می‌دهید مقادیر درست نمایش داده شود؛ پس باید دقیقا با چیزی که خودتان روی روتر تنظیم کرده‌اید یکی باشند. RADIUS Secret اما هم برای سرور RADIUS خود پنل استفاده می‌شود و هم (اختیاری) می‌تواند مستقیما روی روتر پوش شود - ادامه را بخوانید.

**نکته مهم درباره‌ی خروجی OpenVPN:** چون ساخت فایل واقعی `.ovpn` نیاز به embed کردن سرتیفیکیت CA روتر دارد (که پنل بهش دسترسی نداره)، دکمه‌ی «دریافت کانفیگ» برای OpenVPN فقط **آدرس سرور، پورت، یوزرنیم و پسورد** را به‌صورت متن نشان می‌دهد - نه یک فایل `.ovpn` آماده. فایل `.ovpn` نهایی را خودتان یک‌بار (با همان CA که روی روتر ساختید) به‌عنوان قالب آماده کنید و برای هر مشتری فقط یوزر/پسوردش را داخل آن جایگزین کنید. L2TP هم به همین شکل فقط یوزر/پسورد و PSK را نشان می‌دهد چون L2TP معمولا فایل کانفیگ ندارد و از تنظیمات شبکه سیستم‌عامل وارد می‌شود.

### راه‌اندازی RADIUS (برای OpenVPN/L2TP)

پنل یک سرور RADIUS واقعی (auth + accounting، طبق RFC2865/2866) روی همین بک‌اند اجرا می‌کند - نیازی به سرویس RADIUS جدا نیست. پیش‌فرض پورت‌ها: **1812/UDP** برای authentication و **1813/UDP** برای accounting (هر دو در `docker-compose.yml` باز شده‌اند).

**مرحله ۱ - Secret:** در ویرایش نود میکروتیک، یک مقدار برای «RADIUS Secret» بگذارید و ذخیره کنید (یک رشته تصادفی و طولانی، فقط بین پنل و همان روتر مشترک است).

**مرحله ۲ - معرفی پنل به روتر، به یکی از دو روش:**

- **روش خودکار (پیشنهادی):** در همان فرم ویرایش نود، بخش «OpenVPN / L2TP»، آدرس IP این سرور پنل (همانی که روتر باید بهش وصل شود) را وارد کنید و روی «اعمال RADIUS روی روتر» بزنید. پنل با همان اتصال RouterOS API موجود، این دو کار را روی روتر انجام می‌دهد - چیز دیگری لمس نمی‌شود:
  ```
  /radius add service=ppp address=<آدرس پنل> secret=<RADIUS Secret> authentication-port=1812 accounting-port=1813
  /ppp aaa set use-radius=yes accounting=yes interim-update=00:05:00
  ```
- **روش دستی:** همان دو دستور بالا را خودتان در ترمینال/Winbox روتر بزنید.

**محدودیت شناخته‌شده:** فقط **PAP** و **CHAP** پشتیبانی می‌شوند (روش‌های پیش‌فرض PPP در RouterOS)؛ **MS-CHAPv2 پیاده‌سازی نشده است**. اگر پروفایل PPP روتر را طوری تنظیم کرده‌اید که فقط MS-CHAPv2 قبول کند، احراز هویت شکست می‌خورد - در پروفایل PPP گزینه‌های `pap`/`chap` را هم فعال نگه دارید.

**نکته درباره آدرس روتر:** پنل، پکت‌های RADIUS رسیده را با آدرس IP مبدا آن‌ها به لیست نودهای «آدرس IP روتر» تطبیق می‌دهد تا Secret درست را برای هر روتر انتخاب کند. اگر روتر پشت NAT چندلایه یا چند اینترفیس خروجی دارد و آدرس متفاوتی برای اتصال RADIUS استفاده می‌کند، فیلد «آدرس IP روتر» را با همان آدرس واقعی هماهنگ کنید.

> **محدودیت شمارنده:** بر خلاف WireGuard (که شمارنده هر peer تا وقتی وجود دارد ثابت می‌ماند)، مصرف OpenVPN/L2TP آنی و از طریق پکت‌های Accounting (Start/Interim-Update/Stop) دریافت می‌شود، نه polling. اگر روتر ری‌استارت شود یا پکت Stop به هر دلیل گم شود، ممکن است مقدار کمی از حجم جا بماند یا یک‌بار به اشتباه صفر در نظر گرفته شود.

**مصرف نزدیک به لحظه‌ای (near-live):** میزان دقت به‌روزرسانی مصرفِ کاربرِ درحال‌اتصال به فاصله ارسال پکت‌های `Interim-Update` توسط روتر بستگی دارد. در فرم «اعمال RADIUS روی روتر» یک فیلد برای این فاصله وجود دارد (فرمت `HH:MM:SS`، پیش‌فرض روتر ۵ دقیقه) - مقدار کوچک‌تر مثل `00:01:00` باعث می‌شود مصرف کاربرهای متصل هر ۱ دقیقه به‌روز شود، به قیمت کمی ترافیک/بار بیشتر.

### ۲. سرور V2Ray/Xray

دو روش اتصال برای مدیریت نود Xray وجود دارد که موقع افزودن/ویرایش نود از نوع «V2Ray / Xray» قابل انتخاب است:

- **SSH** (پیش‌فرض): پنل مستقیم به سرور Xray با SSH وصل می‌شود و `config.json` را ویرایش می‌کند.
- **پنل 3X-UI** (بدون نیاز به SSH): برای وقتی که Xray/3X-UI جایی نصب شده که SSH بهش دسترسی ندارید (مثلا داخل کانتینر میکروتیک). پنل ما به‌جای SSH، مستقیم با API خودِ وب‌پنل 3X-UI صحبت می‌کند. آدرس وب پنل 3X-UI (شامل مسیر امنیتی که هنگام نصب تعیین کرده‌اید) و شناسه (ID) اینباند مورد نظر را وارد کنید، به‌علاوه یکی از این دو روش احراز هویت:
  - **API Token** (ترجیحی): از تنظیمات پنل → Settings → Authentication → API Token یک توکن بساز و همونو کپی کن.
  - یوزر/پسورد پنل (اگه API Token نداری).

  این حالت هم API جدید (`/panel/api/clients/...`) و هم API کلاسیک (`/panel/api/inbounds/addClient`) رو امتحان می‌کنه - هر کدوم که روی پنل شما جواب بده خودکار انتخاب میشه. فقط با اینباندهای `vless` تست/پشتیبانی می‌شود.

جزئیات زیر مخصوص حالت SSH است:

پنل برای مدیریت کاربران و خواندن آمار مصرف، از طریق **SSH** به سرور Xray وصل می‌شود (نیازی به کامپایل gRPC نیست). دو چیز باید در `config.json` سرور Xray از قبل آماده باشد:

**الف) بخش API + اینباند مخصوص آمار:**

```json
"api": {
  "tag": "api",
  "services": ["HandlerService", "StatsService", "LoggerService"]
},
"stats": {},
"policy": {
  "levels": { "0": { "statsUserUplink": true, "statsUserDownlink": true } },
  "system": { "statsInboundUplink": true, "statsInboundDownlink": true }
},
"inbounds": [
  {
    "tag": "api",
    "listen": "127.0.0.1",
    "port": 10085,
    "protocol": "dokodemo-door",
    "settings": { "address": "127.0.0.1" }
  }
]
```

**ب) اینباند اصلی که کاربران واقعی به آن وصل می‌شوند** باید یک `tag` مشخص داشته باشد (مثلا `proxy`) و لیست `settings.clients` آن (حتی خالی) وجود داشته باشد؛ پنل این آرایه را مدیریت می‌کند.

- سرویس Xray باید با `systemd` مدیریت شود (`systemctl restart xray`).
- کاربر SSH باید دسترسی نوشتن روی فایل کانفیگ و اجرای `systemctl restart` و باینری `xray` (برای `xray api statsquery`) را داشته باشد.
- **مهم:** فرمت خروجی `xray api statsquery` می‌تواند بین نسخه‌های مختلف Xray-core کمی فرق کند. پارسر فعلی (`backend/app/services/xray_client.py`) با فلگ `-json` نوشته شده؛ پیش از استفاده در محیط واقعی، یک‌بار روی سرور خودتان تست کنید.

## نصب سریع (یک دستور - پیشنهادی برای نصب جدید)

یک نصاب شبیه اسکریپت نصب پروژه‌های معروف (مثل 3x-ui) اضافه شده که Docker را (اگر نبود) نصب می‌کند، `.env` را با `SECRET_KEY` تصادفی و رمز ادمین امن می‌سازد، پروژه را بالا می‌آورد، و یک دستور مدیریتی `usermanager` (start/stop/restart/status/logs/update/uninstall) روی سرور نصب می‌کند.

**از روی خود ویندوزتان (ساده‌ترین حالت - جایگزین دستورات دستی zip/scp/ssh):**

```powershell
# کنار پوشه usermanager، فایل‌های install.sh و install_from_windows.ps1 را داشته باشید
.\install_from_windows.ps1 -ServerIp SERVER_IP
# برای پورت دلخواه: .\install_from_windows.ps1 -ServerIp SERVER_IP -Port 8080
# برای آپدیت بعدی همین سرور با کد جدید:  .\install_from_windows.ps1 -ServerIp SERVER_IP -Update
```

این اسکریپت پوشه `usermanager` را (بدون `node_modules`/`.git`/`venv`/دیتا) tar می‌کند، به سرور `scp` می‌کند و `install.sh` را همان‌جا اجرا می‌کند - در یک دستور.

**مستقیم روی سرور (اگر `usermanager.tar.gz` یا `install.sh` را خودتان از قبل آپلود کرده‌اید):**

```bash
sudo bash install.sh              # نصب اولیه یا آپدیت امن (idempotent)
sudo bash install.sh --port 8080  # با پورت پنل دلخواه
sudo bash install.sh uninstall    # حذف کامل
```

منبع سورس به ترتیب اولویت: فایل `usermanager.tar.gz` کنار `install.sh`، یا `USERMANAGER_REPO_URL` (اگر پروژه را روی گیت‌هاب/گیت‌لب خصوصی گذاشته‌اید)، یا پروژه‌ای که از قبل در مسیر نصب موجود است. بعد از نصب، هر وقت خواستید با دستور `usermanager` (بدون آرگومان) یک منوی مدیریتی ساده باز می‌شود.

## ویژگی‌های پنل مدیریت

- **داشبورد:** تعداد کل/فعال/غیرفعال/اتمام-حجم کاربران، تعداد نودها و نودهای آنلاین، تعداد کاربران آنلاین همین لحظه (چه RADIUS چه Xray)، نمودار مصرف کلی ۲۴ ساعت اخیر - همه با کوئری‌های تجمیعی SQL (نه لود کردن همه ردیف‌ها در پایتون)، هر آمار هم روی صفحه کاربران قابل کلیک/فیلتر است.
- **کاربران:** جستجو + صفحه‌بندی سمت سرور، فیلتر بر اساس وضعیت، مرتب‌سازی، نشانگر آنلاین/آفلاین زنده، ساخت تکی (دستی یا با پکیج)، ساخت گروهی (دستی یا با پکیج)، ویرایش گروهی (افزودن حجم/روز، ریست مصرف، فعال/غیرفعال، تغییر سقف اتصال هم‌زمان روی چند کاربر انتخاب‌شده هم‌زمان).
- **جزئیات کاربر:** افزودن/حذف اتصال از هر پروتکل، دریافت کانفیگ/لینک هر اتصال، ویرایش سهمیه/انقضا/موجودی کیف پول/سقف اتصال هم‌زمان، رفع بن.
- **سرورها (نودها):** افزودن/ویرایش نود میکروتیک یا Xray، تست اتصال، اعمال خودکار تنظیمات RADIUS، ایمپورت کاربران موجود (از `/ppp/secret`، از User Manager خودِ میکروتیک، یا از کلاینت‌های موجود روی 3X-UI).
- **پکیج‌ها:** بخش جدا، توضیح کامل در ادامه.
- **آموزش:** بخش جدا، توضیح کامل در ادامه.
- **تنظیمات:** تغییر رمز ادمین، اطلاعات پرداخت کارت‌به‌کارت، کلیدهای API (برای ربات‌های شخص ثالث)، تنظیمات کامل ربات تلگرام داخلی، نصب ربات روی سرور دوم، بک‌آپ دیتابیس.

## سیستم پکیج‌ها

از صفحه «پکیج‌ها» هر تعداد پلن قابل‌فروش تعریف می‌کنید - هرکدام شامل:

- نام، حجم (GB، ۰ = نامحدود)، مدت (روز، خالی = بدون انقضا)، قیمت (تومان).
- سقف اتصال هم‌زمان کلی برای کل پکیج (نه هر سرویس جدا).
- **سرویس‌های همراه (اختیاری):** یک یا چند سرور+پروتکل که با انتخاب همین پکیج به‌طور خودکار برای کاربر ساخته می‌شوند (مثلا یک پکیج می‌تواند هم‌زمان هم یک WireGuard روی سرور A و هم یک Xray روی سرور B بدهد) - دیگر نیازی به انتخاب دستی سرور/پروتکل نیست.
- **فایل‌های ضمیمه (اختیاری):** هر فایلی (کانفیگ آماده، راهنمای نصب، نصاب اپلیکیشن) که بلافاصله بعد از خرید/تمدید این پکیج توسط ربات برای مشتری ارسال می‌شود.
- **متن دلخواه (اختیاری):** پیامی که بعد از خرید/تمدید، همراه اتصالات و فایل‌ها برای مشتری فرستاده می‌شود (مثلا راهنمای پشتیبانی).

پکیج‌ها هم در «ساخت با پکیج» فرم تکی، هم در «ساخت گروهی کاربران» (به‌جای وارد کردن دستی حجم/انقضا/اتصالات)، و هم در فلوی خرید ربات قابل استفاده‌اند.

## کیف پول / اعتبار مشتری

هر کاربر یک فیلد `balance` (تومان) دارد که از دو راه شارژ می‌شود:

1. **کارت‌به‌کارت + تایید ادمین:** مشتری از ربات مبلغ دلخواه یا یکی از مبالغ پیشنهادی (قابل تنظیم از تنظیمات پنل) را انتخاب می‌کند، رسید را می‌فرستد، و ادمین از همان چت تلگرام تایید/رد می‌کند.
2. **از پنل:** مستقیم در صفحه جزئیات کاربر.

از موجودی هم می‌شود مستقیم (بدون تایید ادمین) پکیج خرید/تمدید کرد - «پرداخت از اعتبار» در ربات. کسر موجودی به‌صورت اتمیک (یک `UPDATE` شرطی در سطح دیتابیس، نه خواندن-و-نوشتن جداگانه در پایتون) انجام می‌شود تا دو درخواست هم‌زمان (مثلا دوبار زدن سریع دکمه) نتوانند موجودی را منفی کنند؛ اگر فعال‌سازی سرویس بعد از کسر موجودی به هر دلیل شکست بخورد، مبلغ خودکار به کیف پول برمی‌گردد.

## بخش آموزش

از صفحه «آموزش» هر تعداد راهنما (عنوان + متن + عکس/ویدیوی نامحدود) بسازید - مثلا «نصب WireGuard روی اندروید». این‌ها از منوی «📚 آموزش» ربات برای هر مشتری قابل مشاهده‌اند، بدون نیاز به دخالت ادمین.

## ربات تلگرام داخلی

ربات (بر پایه‌ی aiogram 3) مستقیم داخل همین پروسه بک‌اند اجرا می‌شود - نیازی به سرور/کانتینر جدا، فایل `.env` دستی یا SSH نیست. راه‌اندازی:

1. یک ربات با [@BotFather](https://t.me/BotFather) بسازید و توکن را بگیرید.
2. آیدی عددی تلگرام خودتان را بگیرید (مثلا از [@userinfobot](https://t.me/userinfobot)).
3. از پنل → **تنظیمات → ربات تلگرام**: توکن و آیدی عددی ادمین(ها) را وارد و «ربات فعال باشد» را تیک بزنید، ذخیره کنید.

### بخش مشتری

- `/start` یا دکمه‌های منو: خرید پکیج جدید، تمدید، مشاهده اکانت من، افزایش اعتبار، وصل کردن حساب قدیمی (برای مشتری‌هایی که از قبل در پنل هستند)، آموزش.
- بعد از خرید/تمدید موفق، اتصالات به‌شکل زیبا و مخصوص هر پروتکل فرستاده می‌شود: WireGuard → فایل `.conf` + QR کد؛ Xray/VLESS → لینک قابل کپی + QR کد؛ OpenVPN/L2TP → متن سرور/پورت/یوزر/پسورد (و PSK برای L2TP).
- **منوی دستورات (Menu)** تلگرام هم پر می‌شود: `/start` `/account` `/buy` `/topup` `/tutorials` `/link` `/help` - میانبر برای همان گزینه‌های دکمه‌ای. (اگر بعد از فعال‌سازی این دستورات را در یک چت از قبل باز نمی‌بینید، مشکل معمولا کش خود اپ تلگرام روی آن دستگاه است - چت را ببندید/دوباره باز کنید یا اپ را ری‌استارت کنید؛ سمت سرور در لاگ (`docker compose logs backend | grep "commands menu"`) قابل تایید است.)

### بخش ادمین

- دستورات اضافه در منو: `/newuser` `/users` `/pending` `/broadcast`.
- ساخت/جستجوی سریع کاربر، دیدن/تایید/رد رسیدهای در انتظار (خرید/تمدید/شارژ کیف پول)، پیام همگانی به همه‌ی کسانی که تا حالا از ربات استفاده کرده‌اند.
- تایید/رد یک رسید به‌صورت اتمیک «قفل» می‌شود (یک ستون وضعیت میانی در دیتابیس محلی ربات) تا اگر دو ادمین هم‌زمان (یا یک ادمین با دوبار کلیک سریع) روی یک رسید بزنند، فقط یکی واقعا پردازش شود - دیگری پیام «همین الان توسط ادمین دیگری در حال بررسی است» می‌بیند، نه این‌که کاربر دوبار شارژ/فعال شود.

### معماری فنی (برای توسعه‌دهنده)

- ربات روی یک ترد پس‌زمینه با event loop خودش اجرا می‌شود (شبیه سرور RADIUS)، جدا از event loop اصلی FastAPI.
- `panel_bridge.py` پل بین handlerهای ربات و منطق پنل است؛ در حالت عادی مستقیم و در-پروسه (یک `SessionLocal()` تازه به ازای هر فراخوانی، thread-safe) با دیتابیس صحبت می‌کند - در حالت «نصب روی سرور دوم» به‌طور خودکار به `remote_bridge.py` (همان API که در بخش «API ربات مشتری» مستند شده، با `X-API-Key`) سوییچ می‌کند؛ کد خودِ handlerها در هر دو حالت یکسان است.
- ارسال پیام/فایل از بیرون از حلقه‌ی polling اصلی (مثلا از اسکجولر پس‌زمینه برای یادآوری روزانه یا بک‌آپ) از یک نمونه‌ی کوتاه‌عمر `Bot(token=...)` جدا استفاده می‌کند، نه نمونه‌ی در-حال-polling - چون تلگرام برای هر توکن فقط یک `getUpdates` هم‌زمان قبول می‌کند، ولی ارسال پیام محدودیتی ندارد.

## نصب ربات روی سرور دوم

از پنل → تنظیمات → «نصب ربات روی سرور دیگر»: آی‌پی و رمز روت یک سرور دوم را بدهید، پنل به‌صورت کاملا خودکار (SSH) به آن سرور وصل می‌شود، در صورت نیاز Docker نصب می‌کند، همان سورس/ایمیج بک‌اند را آنجا بالا می‌آورد ولی در «حالت فقط ربات» - یعنی ربات همان‌جا اجرا و polling می‌شود، اما از طریق اینترنت (پورت 8000 این سرور، با `X-API-Key`) همچنان به دیتابیس واقعی همین سرور وصل است.

چون تلگرام برای هر ربات فقط یک نمونه‌ی در حال دریافت پیام قبول می‌کند، با نصب موفق روی سرور دوم، ربات روی همین سرور خودکار متوقف می‌شود؛ با دکمه‌ی «بازگرداندن ربات به همین سرور» می‌توانید برعکسش کنید.

**نکات امنیتی این قابلیت:** رمز SSH فقط در حافظه، برای مدت همان یک درخواست نصب/توقف نگه‌داری می‌شود و هرگز در دیتابیس ذخیره نمی‌شود. فایل `.env` ساخته‌شده روی سرور دوم (حاوی توکن ربات و کلید API پنل) به‌صورت خودکار `chmod 600` می‌شود. کلید میزبان SSH سرور دوم به‌صورت خودکار پذیرفته می‌شود (اولین اتصال، بدون pinning از قبل) - این یک تبادل معمول در ابزارهای «آی‌پی و رمز بده تا نصب کنم» است؛ اگر در شبکه‌ای با ریسک MITM بالا هستید، بعد از اولین نصب فینگرپرینت کلید را دستی تایید کنید.

## بک‌آپ دیتابیس

- **خودکار:** هر ۶ ساعت (۴ بار در روز، ساعت‌های ۰، ۶، ۱۲، ۱۸) کل دیتابیس بک‌آپ گرفته و به تمام ادمین‌های ربات تلگرام (همان لیست تنظیم‌شده در «تنظیمات → ربات تلگرام») ارسال می‌شود.
- **دستی:** از پنل → تنظیمات → «بک‌آپ دیتابیس» → «دریافت بک‌آپ فوری» - همزمان دانلود می‌شود، به تلگرام هم می‌رود، و در لیست بک‌آپ‌های اخیر (روی دیسک سرور، پوشه‌ی `backend/data/backups`) نگه‌داری می‌شود (آخرین ۴۰ فایل، قدیمی‌تر خودکار پاک می‌شود).
- بک‌آپ‌گیری با API رسمی `sqlite3 .backup()` انجام می‌شود (نه کپی فایل خام)، پس حتی وقتی دیتابیس در حال استفاده است هم امن است.

## محدودیت اتصال هم‌زمان (Shared Users) و بن موقت

هر اتصال OpenVPN/L2TP یک «حداکثر تعداد اتصال هم‌زمان» دارد (`max_concurrent_sessions`؛ ۰ یا خالی = نامحدود، پیش‌فرض ۱ - معادل مفهوم `shared-users` در User Manager خودِ میکروتیک). این مقدار موقع ساخت اتصال قابل تنظیم است و بعدا هم از صفحه کاربر قابل ویرایش است. یک سقف در سطح کل کاربر هم وجود دارد (`User.max_concurrent_sessions`) که روی مجموع همه‌ی اتصالات آن کاربر اعمال می‌شود، نه هر اتصال جدا (مثلا کاربری با ۳ سرور از یک پکیج و سقف ۱، فقط می‌تواند هم‌زمان به یکی از آن سه وصل باشد).

- سرور RADIUS پنل، تعداد سشن‌های فعال (بین Accounting Start و Stop) هر اتصال را نگه می‌دارد. وقتی درخواست ورود جدیدی برسد و یوزر/پسورد درست باشد ولی تعداد سشن‌های فعال به سقف رسیده باشد، پنل فقط **همان درخواست جدید را رد می‌کند** - سشن‌های فعال قبلی قطع نمی‌شوند (چون این کار نیاز به فعال بودن CoA/Disconnect-Request روی روتر دارد که پیش‌فرض تنظیم نیست).
- اگر همین اتصال در بازه‌ی ۱ دقیقه، **۵ بار یا بیشتر** با وجود پر بودن ظرفیت تلاش به اتصال بزند (نشانه‌ی اکانت لو رفته/به اشتراک گذاشته‌شده)، به‌مدت **۳ دقیقه** به‌طور کامل بن می‌شود. این بن از صفحه کاربر (دکمه «رفع بن») قابل لغو دستی است.
- سشن‌هایی که پکت `Stop` آن‌ها به هر دلیل گم شود، بعد از ۱۵ دقیقه بدون فعالیت به‌صورت خودکار پاک‌سازی می‌شوند (job زمان‌بندی‌شده هر ۵ دقیقه).
- موقع ایمپورت از User Manager خودِ میکروتیک، مقدار `shared-users` هر اکانت مستقیما به همین `max_concurrent_sessions` منتقل می‌شود.

## ساخت/ویرایش گروهی کاربران و بهینه‌سازی دیتابیس

- **ساخت گروهی:** از صفحه کاربران، دکمه «ساخت گروهی» یک پیشوند + تعداد می‌گیرد و کاربرها را به‌صورت `prefix1، prefix2، ...` می‌سازد (نام‌های تکراری رد می‌شوند و در گزارش نهایی نشان داده می‌شوند). دو حالت دارد: یا یک **پکیج** انتخاب می‌کنید (حجم/انقضا/سرویس‌ها همه از پکیج می‌آید، دقیقا مثل «ساخت با پکیج» تکی)، یا حجم/انقضا/اتصالات را دستی مشخص می‌کنید. حداکثر ۱۰۰۰ کاربر در هر بار.
- **ویرایش گروهی:** با تیک زدن چند کاربر در لیست (یا تیک «انتخاب همه‌ی این صفحه»)، دکمه «ویرایش گروهی» ظاهر می‌شود - افزودن حجم/اعتبار، ریست مصرف، تغییر وضعیت (فعال/غیرفعال) و تغییر محدودیت اتصال هم‌زمان روی همه‌ی انتخاب‌شده‌ها یکجا اعمال می‌شود (حداکثر ۱۰۰۰ کاربر در هر درخواست).
- **صفحه‌بندی و جستجوی سمت سرور:** لیست کاربران همه را یکجا لود نمی‌کند؛ صفحه‌به‌صفحه (۵۰ تایی) و با جستجوی سمت سرور (`GET /api/users?page=&page_size=&search=`) کار می‌کند. ایندکس‌های دیتابیس روی `ppp_username`، `wg_peer_name`، `xr_email`، `users.username`، و کلیدهای خارجی پرترافیک (`connections.user_id/node_id`، `usage_logs.user_id/connection_id`، `package_files.package_id`، `tutorial_media.tutorial_id`، `package_connections.package_id/node_id`) اضافه شده‌اند.
- **آمار داشبورد** با کوئری‌های تجمیعی SQL (`COUNT`/`SUM`/`GROUP BY`) محاسبه می‌شود، نه با لود کردن همه‌ی کاربران/نودها در پایتون - با رشد تعداد کاربران هم سریع می‌ماند.

## جمع‌بندی منطق سهمیه مشترک

1. هر ۳۰ ثانیه (قابل تنظیم با `POLL_INTERVAL_SECONDS`)، پنل به هر نود سر می‌زند.
2. برای WireGuard: شمارنده‌های `rx`/`tx` هر peer از میکروتیک خوانده می‌شود.
3. برای OpenVPN/L2TP: مصرف polling نمی‌شود - همان لحظه که پکت‌های RADIUS Accounting (Interim-Update/Stop) از روتر می‌رسند، پردازش می‌شوند.
4. برای Xray: از طریق SSH یا 3X-UI API، مصرف همه کاربران خوانده می‌شود.
5. اختلاف (دلتا) نسبت به مقدار قبلی محاسبه و به `used_bytes` **کاربر** (نه فقط آن اتصال) اضافه می‌شود.
6. اگر `used_bytes >= total_quota_bytes` یا تاریخ انقضا گذشته باشد، تمام اتصالات آن کاربر هم‌زمان غیرفعال می‌شوند.
7. با ریست حجم یا افزایش سهمیه از پنل، اتصالات به‌صورت خودکار دوباره فعال می‌شوند.

## API ربات مشتری (برای اتصال به ربات/سیستم فروش شخص ثالث)

این همان API‌ای است که ربات داخلی خودِ پروژه هم از آن استفاده می‌کند (چه در-پروسه، چه از سرور دوم) - پس کاملا feature-complete و همیشه به‌روز است. جدا از پنل ادمین است و با یک کلید ثابت کار می‌کند، نه لاگین کاربر/رمز. کلید را از صفحه **تنظیمات پنل → «کلیدهای API»** بسازید.

- آدرس پایه: `http://SERVER_IP:8000/api/bot` (یا `http://SERVER_IP/api/bot` اگر پشت همان nginx فرانت هستید)
- هدر لازم روی همه‌ی درخواست‌ها: `X-API-Key: <کلید ساخته‌شده>`

### ساخت کاربر جدید (به همراه اتصال روی یک یا چند سرور)

```bash
curl -X POST http://SERVER_IP:8000/api/bot/users \
  -H "X-API-Key: YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "username": "customer_123",
    "full_name": "مشتری تلگرام",
    "quota_gb": 50,
    "expire_days": 30,
    "connections": [
      { "node_id": 1, "protocol": "wireguard" },
      { "node_id": 2, "protocol": "xray" }
    ]
  }'
```

مقدار `protocol` یکی از `wireguard` / `openvpn` / `l2tp` / `xray` است. شناسه‌ی `node_id` را از `GET /api/bot/nodes` بگیرید. پاسخ شامل اطلاعات کامل کاربر و برای هر اتصال هم یک `config_text` خوانا و هم فیلدهای ساختاریافته (`server`/`port`/`username`/`password`/`psk`) است تا ربات‌تان بتواند پیام مخصوص هر پروتکل بسازد.

### سایر endpointهای کلیدی

| متد | مسیر | کاربرد |
|---|---|---|
| `GET` | `/api/bot/nodes` | لیست سرورهای فعال |
| `GET` | `/api/bot/packages` | لیست پکیج‌های فعال (همراه سرویس‌ها و فایل‌ها) |
| `GET` | `/api/bot/payment-info` | اطلاعات کارت‌به‌کارت |
| `GET` | `/api/bot/tutorials` | لیست آموزش‌های فعال |
| `GET` | `/api/bot/packages/{id}/files/{file_id}/download` | دانلود یک فایل ضمیمه پکیج (بایت خام) |
| `GET` | `/api/bot/tutorials/{id}/media/{media_id}/download` | دانلود یک عکس/ویدیوی آموزش (بایت خام) |
| `GET` | `/api/bot/telegram-user-ids` | همه آیدی‌های تلگرام لینک‌شده (برای پیام همگانی) |
| `POST` | `/api/bot/users/{username}/connections` | افزودن یک اتصال جدید به کاربر موجود |
| `GET` | `/api/bot/users/{username}` یا `/api/bot/users/by-telegram/{telegram_id}` | مشاهده وضعیت/مصرف کاربر |
| `POST` | `/api/bot/users/{username}/renew` | تمدید (افزایش حجم/روز، با یا بدون ریست مصرف) |
| `POST` | `/api/bot/users/{username}/reset-usage` | ریست مصرف بدون تمدید |
| `POST` | `/api/bot/users/{username}/set-enabled` | فعال/غیرفعال کردن کاربر (و همه اتصالاتش روی سرورها) |
| `POST` | `/api/bot/users/{username}/add-balance` | افزایش (یا با مقدار منفی، کسر اتمیک) موجودی کیف پول |
| `POST` | `/api/bot/users/{username}/link-telegram` | وصل کردن یک آیدی تلگرام به یک کاربر موجود |
| `DELETE` | `/api/bot/users/{username}` | حذف کامل کاربر (همه‌ی اتصالاتش هم از روی سرورها حذف می‌شود) |

مستندات تعاملی کامل همه‌ی endpointها (هم پنل ادمین، هم بخش `/api/bot`) در آدرس `http://SERVER_IP:8000/docs` (Swagger UI) هم در دسترس است.

> **نکته امنیتی درباره `link-telegram`:** این endpoint فقط چک می‌کند که آیدی تلگرام مقصد قبلا به کاربر دیگری وصل نشده باشد - تایید مالکیت واقعی نام‌کاربری (مثلا با یک کد یک‌بارمصرف یا تایید دستی ادمین) به عهده‌ی ربات/سیستمی است که این endpoint را صدا می‌زند. اگر ربات شما نام‌کاربری‌های قابل‌حدس (مثل `tg<آیدی_عددی>`) تولید می‌کند، این نکته را قبل از استفاده در محیط واقعی حتما در نظر بگیرید.

## نکات امنیتی

- **`SECRET_KEY` و `DEFAULT_ADMIN_PASSWORD` را حتما در `backend/.env` عوض کنید** - اگر عوض نکنید، هر بار بالا آمدن سرور یک هشدار در لاگ چاپ می‌شود، ولی سرور همچنان (عمدا) بالا می‌آید تا یک دیپلوی موجود قفل نشود. مقدار پیش‌فرض `SECRET_KEY` عمومی/شناخته‌شده است - هرکس آن را بداند می‌تواند توکن ادمین جعل کند.
- پنل را پشت HTTPS (مثلا با nginx + Let's Encrypt) قرار دهید، چون رمزهای میکروتیک/SSH و کلیدهای API داخل دیتابیس پنل نگه‌داری می‌شوند و توکن ادمین هم بدون HTTPS در شبکه قابل شنود است.
- دسترسی کاربر API میکروتیک و کاربر SSH سرور Xray را به حداقل لازم محدود کنید.
- کلیدهای API ربات را مثل رمز عبور مخفی نگه دارید؛ هرکس این کلید را داشته باشد می‌تواند کاربر بسازد/حذف کند/موجودی کیف پول را تغییر دهد.
- **موارد زیر عمدا اتصال به میکروتیک/3X-UI/سرور دوم را بدون تایید کامل گواهی TLS یا کلید میزبان SSH انجام می‌دهند** (چون اکثر نودهای VPN شخصی از گواهی self-signed استفاده می‌کنند و pinning دستی برای این پروژه over-engineering بود): اتصال API-SSL میکروتیک، اتصال HTTP به پنل 3X-UI، و SSH نصب ربات روی سرور دوم. اگر نودهای شما در شبکه‌ای با ریسک واقعی MITM قرار دارند (مثلا اینترنت عمومی بدون VPN بین پنل و نود)، این نکته را در نظر بگیرید.
- کیف پول: کسر موجودی هنگام خرید از اعتبار به‌صورت اتمیک (شرطی در سطح دیتابیس) انجام می‌شود تا race condition/دوبار-کلیک نتواند موجودی را منفی کند یا سرویس رایگان بدهد؛ تایید/رد رسید توسط ادمین هم به همین شکل قفل می‌شود تا دو ادمین نتوانند یک رسید را دوبار پردازش کنند.
- رمز عبور ادمین اولیه دیگر در لاگ چاپ نمی‌شود (فقط یوزرنیم) - اگر لاگ‌های سرور را جایی به اشتراک می‌گذارید (مثلا برای پشتیبانی)، این ریسک قبلا برطرف شده است.

## آپدیت از نسخه قدیمی‌تر / تغییرات دیتابیس

جدول‌ها و ستون‌های جدید (`radius_active_sessions`، `packages`/`package_connections`/`package_files`، `tutorials`/`tutorial_media`، `bot_settings`، `connections.max_concurrent_sessions`/`banned_until`، `users.telegram_id`/`balance`/`max_concurrent_sessions`، ایندکس‌های جدید و غیره) با `Base.metadata.create_all()` فقط برای **جدول‌های کاملا جدید** به‌صورت خودکار ساخته می‌شوند - SQLAlchemy جدول از قبل موجود را alter نمی‌کند. برای هر آپدیتی که یک **ستون جدید** به جدول **موجود** اضافه می‌کند (مثلا زمانی که این README به‌روز شد، آخرین مورد افزودن ایندکس‌ها روی `connections`/`usage_logs`/`package_files`/`package_connections`/`tutorial_media` بود)، باید دستی روی دیتابیس در حال اجرا این کار را انجام دهید - یا با `ALTER TABLE`، یا (برای ایندکس‌ها) با یک یک‌خطی مشابه این از داخل کانتینر:

```bash
docker compose exec -T backend python3 -c "
from app.database import engine
from app import models
models.Base.metadata.create_all(bind=engine)  # فقط جدول‌های کاملا جدید را می‌سازد
with engine.connect() as conn:
    # نمونه: ساخت ایندکس روی یک ستون موجود که قبلا ایندکس نداشت
    conn.execute('CREATE INDEX IF NOT EXISTS ix_connections_user_id ON connections (user_id)')
    conn.commit()
"
```

اگر از یک نسخه‌ی خیلی قدیمی (قبل از این‌همه فیچر) مستقیم آپدیت می‌کنید و برایتان مهم نیست دیتای فعلی نگه داشته شود، ساده‌ترین راه حذف `backend/data/usermanager.db` قبل از بالا آوردن نسخه جدید است (و بازسازی/ایمپورت دوباره‌ی کاربرها).

## عیب‌یابی

- **لاگ بک‌اند:** `docker compose logs -f backend` - شامل لاگ RADIUS، polling مصرف، و ربات تلگرام.
- **ربات فعال نمی‌شود / منوی دستورات خالی است:** لاگ را برای خط‌های `bot commands menu set for default scope` بگردید - اگر این خط با موفقیت چاپ شده، مشکل سمت سرور نیست، احتمالا کش چت تلگرام روی دستگاه مشتری است (چت را ببندید و دوباره باز کنید).
- **مصرف OpenVPN/L2TP آپدیت نمی‌شود:** مطمئن شوید `/radius` و `/ppp aaa` واقعا روی روتر تنظیم شده‌اند (بخش «راه‌اندازی RADIUS») و پورت‌های UDP 1812/1813 از فایروال باز هستند.
- **اتصال Xray برقرار نمی‌شود:** با دکمه «تست اتصال» روی همان نود شروع کنید؛ خطای دقیق (SSH/3X-UI) همان‌جا نمایش داده می‌شود.

## توسعه‌های پیشنهادی آینده

فهرست کامل و اولویت‌بندی‌شده در پیام جداگانه‌ای که همراه همین بروزرسانی README ارسال شد آمده - خلاصه‌ی موارد مهم‌تر:

- افزودن تایید دومرحله‌ای برای «وصل کردن حساب قبلی» در ربات (در حال حاضر فقط چک می‌کند که آیدی تلگرام مقصد آزاد باشد، نه این‌که واقعا صاحب آن نام‌کاربری باشید).
- گزینه‌ی pinning گواهی/کلید میزبان برای اتصالات میکروتیک API-SSL، 3X-UI، و SSH نصب ربات روی سرور دوم (در حال حاضر عمدا غیرفعال‌اند - بالا توضیح داده شده).
- یک تست خودکار (CI) حداقلی برای مسیرهای پرریسک‌تر (احراز هویت، کیف پول، بک‌آپ).
- مهاجرت از SQLite به PostgreSQL برای نصب‌های با تعداد کاربر بسیار بالا (فعلا با ایندکس‌گذاری فعلی برای اکثر نصب‌ها کافی است).
